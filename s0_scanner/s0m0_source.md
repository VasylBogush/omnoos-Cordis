# С0.М0 — Источник (Source)

> **Адрес:** 000–009
> **Тезис:** Куда подключаются источники, чтобы передать сырое — ОМ как MCP-сервер, источник как MCP-клиент.

## Что делает

ОМ работает как **MCP-сервер**. Источник (sensor, satellite-feed, HuggingFace dataset stream,
лабораторное оборудование, чужая система) — **сам подключается к ОМ** как MCP-клиент и
передаёт сырое. M0 — **серверный endpoint С0** и набор обработчиков входящих подключений.

Это **inbound** архитектура (не ОМ ходит за данными, а данные приходят в ОМ). Это входная
точка С0 для непокрытых Gemini модальностей.

## 10 узлов — эскиз, обсудим

(переименование в MCP-server семантике; имена узлов n3, n7, n8 ещё под вопросом — см. анализ ниже)

| # | Адрес | Имя | Тезис простыми словами |
|---|---|---|---|
| n0 | 000 | **Source Registry** | Реестр всех когда-либо подключавшихся источников — кто, чем, статус. |
| n1 | 001 | **Source Registration** | Источник представляется при первом подключении: тип, адрес, описание. |
| n2 | 002 | **Authentication** | Источник аутентифицируется в ОМ: API key, mTLS, OAuth-flow. |
| n3 | 003 | **Connection Accept** | Входящее MCP-соединение принимается (handshake, session start). |
| n4 | 004 | **Receive** | Runtime приём MCP-сообщений с сырыми чанками. |
| n5 | 005 | **Parse** | Декодирование MCP-payload в native-формат модальности (GeoTIFF, FASTA, EDF, etc). |
| n6 | 006 | **Place of the Other ⚡** | Местный полюс Иного: source-side ошибки или непредсказуемое в потоке. |
| n7 | 007 | **Buffer & Throttle** | Управление потоком: очередь, rate limits, backpressure от downstream. |
| n8 | 008 | **Health Monitor** | Heartbeat, latency, error rate per источник. |
| n9 | 009 | **Forward** | Передача дальше: `{raw_chunk, source_metadata}` → M1 (lookup) + M5 (Архив параллельно). |

## Важное про каскад n6 — не в С9!

Я был неточен в исходной формулировке. **М0.n6 = 006** даёт только **одиночный carry**:
'6'→'3' carry; '0'→'9' break. Результат: **093 = С0.М9.n3** (Wanderer узел 3) **внутри С0**.

В С9 каскадирует **только центральный полюс Иного С0**: **С0.М6.n6 = 066** → двойной carry
до сервиса: '6'→'3' carry; '6'→'3' carry; '0'→'9' break → **933 (С9.М3.n3)**.

**Структурный паттерн:** каждый локальный n6 в модулях М0..М5,М7..М9 идёт **внутрь С0**;
только М6.n6 выходит в С9. Это значит, что **сигнал "не справился источник"** не идёт
сразу в С9 — он проходит через **локальный обработчик внутри С0** (Wanderer.n3 в случае
М0.n6).

## Архитектурный принцип n3 = приёмник Иного

**n3 в каждом модуле = приёмник Иного от другого модуля С0**, по обратной JUMP_MAP:

| Куда (n3 модуля X) | Откуда (n6 модуля Y, JUMP_MAP[Y]=3·) |
|---|---|
| М0.n3 (003) ← | М3.n6 (036) Projector Workshop |
| М1.n3 (013) ← | М7.n6 (076) Bricolage |
| М2.n3 (023) ← | М4.n6 (046) Translation |
| М3.n3 (033) ← | М0.n6 (006)? — нет, 006 → 093, не 033 |

Поправка: n3 принимает Иное от модуля Y, где **JUMP_MAP[Y]·"3"** не сходится напрямую.
Точный обратный mapping:

- М0.n3 (003) — приёмник n6 от модуля X где `JUMP_MAP[X]=0`: это X=**3** (JUMP_MAP['3']=0). То есть М3.n6 → М0.n3. ✓
- М1.n3 (013) — приёмник от X где JUMP_MAP[X]=1: X=**7**. М7.n6 → М1.n3. ✓
- М2.n3 (023) ← М4.n6 (JUMP_MAP['4']=2). ✓
- ...

Это структурно: **n3 каждого модуля = "сюда приходит Иное от соседнего модуля"**. В моём
эскизе n3 = "Connection Accept". Это **не учитывает** двойную роль n3 как приёмника от
M3.n6 (Workshop). Возможно надо переименовать n3 (или признать что n3 совмещает две роли —
штатный приём соединений + обработка ошибок Workshop).

## Где переходы flag=0 работают разумно

- n2 → n8 (Auth fail → Monitor зафиксирует) ✓
- n3 → n0 (Connection сорван → Registry обновить) ✓
- n4 → n2 (Receive не идёт → re-auth) ✓
- n5 → n7 (Parse не удался → Buffer для retry) ✓
- n9 → n6 (Forward не удался → в Иное) ✓
- n6 → 093 (Иное → Wanderer.n3 локальный выход) ✓ структурно

## Где переходы flag=0 натянуты

- **n0 → n9** (Registry → Forward): «реестр не нашёл → выдать как есть» — слабо. **n0** в моём именовании = просто реестр, может не иметь смыслового flag=0.
- **n1 → n4** (Registration → Receive): «не нужна регистрация → сразу к приёму» — допустимо, но не сильно.
- **n7 → n1** (Buffer → Registration): «переполнение → к регистрации» — натянуто. Надо думать.
- **n8 → n5** (Monitor → Parse): «monitor → parse» — натянуто. Надо думать.

## Что отсюда следует — открытые архитектурные решения

1. **Имена узлов подбираются под JUMP_MAP, не наоборот.** Это архитектурный принцип ОМ.
   Где flag=0 натянут — имя узла **возможно надо менять**.
2. **n3 в каждом модуле = двойная роль:** штатный приём (входной поток в средине модуля)
   + приёмник Иного от соседнего модуля.
3. **М0.n6 каскад НЕ в С9**, а в **С0.М9.n3** — Wanderer. Это меняет понимание потока ошибок.
4. **Каскад в С9 — только из М6.n6 = 066** (центральный полюс Иного С0).
5. Узлы n7, n8 в моём эскизе — кандидаты на переименование (flag=0 натянут).

## Поток runtime

Штатный путь одного входящего MCP-сообщения (flag=1 по умолчанию):
`n3 (Connection) → n4 (Receive) → n5 (Parse) → n6 (Place of the Other, прошёл) → n7 (Buffer) → n8 (Monitor) → n9 (Forward) → M1.n0`.

Ошибочный путь источника:
`сбой на n3/n4/n5 → flag=0 → jump по таблице → ... → n9 (Forward) → flag=0 → n6 → 093 (Wanderer.n3)`.

## Открытые вопросы

1. **Гранулярность «источника»** — это endpoint URL, API account, физическое устройство, или
   логический канал? Один источник может выдавать несколько модальностей?
2. **Где хранить credentials (n2)** — vault, env-переменные, отдельная таблица в БД, секрет-сервис?
3. **Stream vs batch** — один и тот же интерфейс n4 Receive, или разные узлы для разных моделей
   доставки?
4. **Modality detection** — определяется здесь (по MCP-headers), или явное указание от источника,
   или это задача M1 Catalog после Forward?
5. **Failover** — если источник отключился: retry-policy, dead-letter, переключение на бэкап?
6. **Связь с M5 (Архив)** — все ли raw chunks идут в M5 параллельно с n9 Forward, или только те,
   что не прошли parse?
7. **Переименование n3, n7, n8** под более осмысленные flag=0 переходы — или признать, что
   некоторые узлы редко возвращают flag=0 в реальной работе.
8. **MCP-протокол сам** — какая версия используется ОМ (MCP 1.x от Anthropic), какие методы
   нужны для С0 (`tools/call`? `sampling`? кастомные?).

## Что не сюда

- **Lookup модальности** в реестре — это M1 Catalog (вход после Forward).
- **Foundation inference** — M2 Foundation Park (далее по потоку).
- **Калибровка перевода** — M8 Calibrator.
