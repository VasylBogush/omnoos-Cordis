# С0 — Сканер

> **Адресация:** 000–099 (100 узлов = 10 модулей × 10 узлов)
> **Зона:** переводчик нестандартных для Gemini модальностей **в текстовое описание** (текст — нативный формат Gemini; embedding делается в С3)
> **Полярная пара:** С9 (Со-Бытие) — кольцо приёма-проявления, **односторонняя** (нет ОС от С9)
> **Статус:** рабочий эскиз, наполняется в процессе
> **Дата эскиза:** 2026-05-24 (правки от 2026-05-24 вечер: выход — текст; развилка на MCP-слое; спектр степеней обработки)

## Что делает С0

Принимает сырой сигнал в **модальностях, не покрытых нативно Gemini Embedding 2** — спутник, EEG,
белок, time series, point cloud и т.п. — и выдаёт **текстовое описание** этого сигнала.
Текст становится нативным форматом для дальнейшей обработки Gemini; embedding откладывается
до С3, где он реально нужен для анализа.

Описание формируется одним из двух путей в зависимости от **степени обработки модальности**
(см. раздел «Спектр степеней» ниже):

- **Embedding-only foundation** (LaBraM, ESM-2, Prithvi, MOMENT, …): спец-embedding +
  векторная близость к **якорям** (эталонные сигналы с известным описанием) +
  LLM-склейка финального текста.
- **Multimodal LLM** (Thought2Text, ProteinGPT, STELLA, MedTsLLM, GeoChat, PointLLM, …):
  прямой `describe(signal) → text` через API или локальный inference.

Нативные для Gemini модальности (text, image, video, audio, PDF) **минуют С0** и идут прямо
в С3 через MCP-инструмент 1 (см. ниже).

## Архитектура подключения

**ОМ = MCP-server с двумя инструментами:**

- **MCP-0 (`translate_other`):** для **нестандартных** Gemini модальностей → вход в С0 (000).
- **MCP-1 (`recognize_native`):** для **стандартных** Gemini модальностей → прямо в С3 (333),
  **минуя С0**.

**Развилка короткого / длинного пути происходит на MCP-слое:** выбор инструмента определяется
типом модальности источника. Это и есть «предпонимание» в смысле OMNOOS — наличие готового
канала Gemini для данной модальности. Если канал есть (степень 3 в спектре) → короткий путь
в С3. Если нет → длинный путь через С0 (текстовое описание) → С1 (длинный путь предподготовки)
→ С2.

Источники **сами подключаются к ОМ** как MCP-клиенты. ОМ не ходит за данными — данные приходят
в ОМ. Это inbound-архитектура, реализованная в M0 (Source) как серверный endpoint + обработчики
входящих подключений.

**Авторизация — внешний слой ОМ**, не внутри С0. Аутентификация — на уровне MCP transport.

## Контракт

### Вход
- Inbound MCP-подключение от источника.
- Сырой сигнал в непокрытой Gemini модальности.
- (опционально) метаданные о модальности и источнике в MCP-сообщении.

### Выход
- **Штатный (099,1 → 100):** в С1.М0 — `{text_description, modality, source_metadata, confidence, spec_embedding?}` (text — основное содержание; spec_embedding опционально как метаданные, если C2/C3 потребуется точная геометрия).
- **Каскад Иного в С9 (066,0 → 933):** ТОЛЬКО из центрального полюса Иного С0 — узла
  С0.М6.n6 = 066 (двойной carry до Сервиса). Локальные n6 модулей М0..М5, М7..М9 — каскадируют
  **внутри Сервиса 0** через JUMP_MAP, не в С9.

## Спектр степеней обработки модальности

Модальности живут в **спектре**, и любая модальность может **мигрировать** по нему во времени
по мере появления новых моделей. С0 устроен под спектр и под миграцию, не под фиксированный
набор моделей.

| Степень | Что есть | Путь к тексту | Где | Примеры |
|---|---|---|---|---|
| **0. Холодная** | foundation моделей нет | ручные якоря или **прямо в M6 → C9** | внутри С0 (M6) или вне | MT-сигнал (Bandyopadhyay/Cantero — точечно) |
| **1. Embedding-only** | спец-foundation модель даёт embedding | спец-embedding + якоря + LLM-склейка | внутри С0 (M2/M3/M4) | LaBraM (EEG), ESM-2 (белок seq), Prithvi (спутник), MOMENT (time series), PointBERT (point cloud) |
| **2. Multimodal LLM** | модель умеет `modality → text` напрямую | прямой `describe()` через API | внутри С0 (M2/M4) | Thought2Text (EEG), ProteinGPT, STELLA (белок), MedTsLLM (биосигналы), GeoChat (спутник), PointLLM (point cloud) |
| **3. Native Gemini** | Gemini Embedding 2 покрывает нативно | минует С0 целиком через MCP-1 | в С3 | text, image, video, audio, PDF |

**Миграция модальности по спектру** (например, степень 1 → 2 при появлении multimodal LLM для
модальности) — должна быть прозрачна для рантайма С0. M1 Каталог хранит **текущую степень**
+ describer-стратегию; M3 Мастерская переключает describer без переписывания архитектуры.

## Поток сигнала

- **Известная модальность степени 1 (embedding-only):** M0 → M1 (lookup: степень 1, якорный describer) → M2 (загрузка spec foundation) → M3 (якорный корпус + промпт склейки) → M4 (`signal → spec_embedding → k-NN по якорям → склейка текста`) → M8 (gate текста) → M9 → С1.М0.
- **Известная модальность степени 2 (multimodal LLM):** M0 → M1 (lookup: степень 2, прямой API) → M2 (адаптер к multimodal LLM) → M4 (`describe(signal) → text`) → M8 (gate текста) → M9 → С1.М0.
- **Новая модальность (нет в Каталоге):** M0 → M1 (not found) → M5 (в архив сырое) → **M6 ⚡ → С9** (односторонне). Параллельно — M7 (бриколаж): попытка собрать describer (якорный корпус с нуля или новый промпт-адаптер).
- **Степень 0 (foundation нет вообще):** M0 → M1 (степень 0) → M5 (архив) → **M6 ⚡ → С9** напрямую.
- **Слабый текст:** M4 → M8 (низкая уверенность / нечитаемо / галлюцинация) → M5 (в архив) + M6 ⚡ → С9.

## Методология создания эскиза модуля

**Порядок прежде логики.** Имена узлов подбираются под JUMP_MAP, не наоборот. Шаги:

1. **Скелет переходов.** Считаем `calc_next_id(addr, flag)` для всех 10 узлов модуля
   (flag=1 и flag=0). Получаем матрицу переходов. Она инвариантна — определяется только
   адресом, не семантикой.

2. **Граничные / структурные узлы первыми:**
   - **n3** (вход с двойной ролью): принимает Иное от соседнего модуля (через carry-механизм),
     плюс работает как штатный шаг потока. Адресован специально — JUMP_MAP['6']=3 carry.
   - **n0** (вход): начальная точка модуля; статический реестр / справочник.
   - **n9** (выход): передаёт дальше в следующий модуль (NN9,1 → следующий MMN).
   - **n6** (Место Иного): местный полюс; flag=0 даёт jump за пределы линейного потока.
     Локально внутри Сервиса (для M6 центрального — выход в С9).

3. **Внутренние узлы цикла коррекций** (1, 2, 4, 5, 7, 8) — в последнюю очередь:
   - Образуют замкнутый цикл по flag=0: `1→4→2→8→5→7→1`.
   - Каждый узел имеет «корректирующего соседа», к которому идёт при flag=0.
   - Имена подбираются так, чтобы пара (узел, его корректор) читалась осмысленно.

4. **Имена** — последний шаг. Описывают **функцию** узла так, чтобы все flag=0 переходы
   давали разумную семантику. Если переход «натянут» — имя надо менять, не схему.

5. **Проверка** на полном цикле потока (flag=1 по умолчанию: n0 → n1 → ... → n9) и
   на типичных аварийных путях (flag=0 на каждом узле).

Эту же методологию применять к M1..M9 при их детализации.

## Контракт с С9 — односторонний

Каскад **066,0 → 933 — единственный путь из С0 в С9**, через центральный полюс Иного.
**Без обратной связи**. С9 хранит непонятое как непонятое, через сообщество
(Polius Inogo, VL, Wright). С0 параллельно может пытаться собрать новый describer (M7 Бриколаж),
но **не по возврату из С9**, а по внутреннему триггеру.

Два разных ответа на «не знаю»:
- С9 — непонятое остаётся как непонятое, проявляется в общем месте сознания.
- С0 (M7) — попытка описать через новый describer (якорный корпус с нуля или новый
  промпт-адаптер к multimodal LLM).

Оба могут срабатывать, или один, или другой. Недоминация.

## Контракт с С1

`C0.M9 (Странник) → C1.M0` через штатное 099,1 → 100.

**Содержание:** `{text_description, modality, source_metadata, confidence, spec_embedding?}`.
text_description — основное; spec_embedding опционально как метаданные.

**С1 = длинный путь предподготовки к С2** (отдельный Сервис, не пустой слот): внимание
фокусируется, объект собирается в целостный образ, идёт поиск узнавания, достраивается
новое предпонимание. Подробнее — `s1_*` (в работе).

## Архитектура развёртывания (черновик)

- **Лёгкая часть** (MCP-роутер, реестр модальностей, склейка текста, каскад): `omnoos-backend/s0_scanner/` на Railway.
- **Foundation inference** (embedding-only foundation модели — тяжёлая GPU-часть): Modal / HuggingFace Inference / возможно Gonka (БЛ) в будущем.
- **Multimodal LLM API** (Thought2Text, ProteinGPT, MedTsLLM, …): внешние API или self-hosted на той же Modal/HF.
- **Якорные корпуса**: object storage (S3/R2) для эталонных сигналов; embedding кеш для якорей.

См. также: «Где может жить С0» — обсуждение 2026-05-24.

## С0 как LLM-зона — уточнение

Формулировка спеки OMNOOS «Сервисы 0, 1, 2 — LLM-зона» в новом прочтении буквальна:
**С0 выдаёт текст**. Описание формируется либо склейкой через LLM (якорный путь), либо
прямым вызовом multimodal LLM. Тем не менее «LLM-зона» не означает «каждый узел реализуется
промптом». Внутри Сервиса есть:

- **Инфраструктурные модули** (М0 Источник) — преимущественно код (Python + MCP SDK + сеть).
  LLM-вставки только в гибридных узлах (n1 нестандартная шапка, n3 семантическое сопоставление
  модальности, n6 семантическая аномалия, n8 мягкая политика).
- **LLM-центричные модули:**
  - **М3 Мастерская describer'ов** — курирование якорных корпусов, авторинг промптов
    склейки, адаптеры к multimodal LLM API.
  - **М4 Перевод** — runtime LLM-склейка или вызов multimodal LLM.
  - **М7 Бриколаж** — поиск foundation модели/multimodal LLM для новой модальности,
    автоматический подбор anchor-данных, тестирование describer'а.
  - **М6 Место Иного С0** — семантическая аномалия, упаковка непонятого в каскад 066,0 → 933.

Это меняет ожидания: М0 ≠ «весь промптом». М3/М4/М7 ≠ «весь код».

- **М5 Архив непрочитанного (050–059).** Хранение всего входящего до различения (сырое + промежуточные тексты). Приёмник Иного от М8 (текст не прошёл gate).
- **М6 Место Иного С0 (060–069).** Центральный полюс Иного Сервиса. **М6.n6 = 066 → 933** — единственный путь в С9. Сюда же — модальности степени 0. **Надо думать.**
- **М7 Бриколаж (070–079).** Полуавтоматическое создание нового describer'а для модальности (якорный корпус с нуля или новый промпт-адаптер). LLM-центричный. Приёмник Иного от М5 (Архив). **Надо думать.**
- **М8 Калибратор (080–089).** Выходной gate качества **текста**: читаемость, фактичность, длина, уверенность. Приёмник Иного от М2 (Foundation Park).
- **М9 Странник (090–099).** Передача в С1.М0 через 099,1 → 100 — **текст** + метаданные. Приёмник Иного от М0 (Источник).

## Глобальные открытые вопросы по С0

### Архитектура

1. **MCP-конфигурация двух инструментов** — реализуем как два tools одного MCP-сервера ОМ, или как два отдельных MCP-сервера?
2. **Авторизация во внешнем слое** — где живёт (С5 Сталкер? Отдельный auth-сервис?), как связана с MCP transport-level аутентификацией?
3. ~~**Объединение С0+С1**~~ — **РЕШЕНО (24.05 вечер):** С1 = длинный путь предподготовки к С2, отдельный Сервис. С0 → С1 контракт = текст + метаданные.
4. **М6 Место Иного С0** — детализация архитектурно открыта; нужен подход к «непонятому» (включая модальности степени 0).
5. **М7 Бриколаж** — триггер запуска без ОС от С9; кто человек в петле; уровень автоматизации.
6. **Спектр степеней обработки модальности** — как M1 Каталог хранит и обновляет степень? Версионирование стратегий describer'а?
7. **Миграция модальности по спектру** (1→2 при появлении multimodal LLM для модальности) — кто инициирует? Как переключение прозрачно для рантайма (откат на старый describer если новый деградирует)?

### Контракты межсервисные

1. **С0 → С1 (099,1 → 100)** — формат содержания зафиксирован (`{text_description, modality, source_metadata, confidence, spec_embedding?}`); требования к структуре/длине/формату текста — TBD.
2. **С0 → С9 (066,0 → 933)** — формат каскада «непонятого» в Со-Бытие.
3. **С0 ↔ С3** — нативные модальности минуют С0 и идут в С3.М3.n3 = 333 через MCP-1. Реализация — один MCP-server с двумя tools, или два отдельных server'а?

### LLM-зона

1. **М3, М4, М6, М7 как LLM-центричные** — реализуются как LLM-агенты с tools или классический код + LLM-вставки?
