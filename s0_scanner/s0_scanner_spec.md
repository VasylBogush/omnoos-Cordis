# С0 — Сканер

> **Адресация:** 000–099 (100 узлов = 10 модулей × 10 узлов)
> **Зона:** переводчик непокрытых Gemini-модальностей в общее embedding-пространство 768D
> **Полярная пара:** С9 (Со-Бытие) — кольцо приёма-проявления, **односторонняя** (нет ОС от С9)
> **Статус:** рабочий эскиз, наполняется в процессе
> **Дата эскиза:** 2026-05-24

## Что делает С0

Принимает сырой сигнал в **модальностях, не покрытых нативно Gemini Embedding 2** — спутник, EEG,
белок, time series, point cloud и т.п. — и выдаёт **embedding в Gemini 768D**.

Нативные для Gemini модальности (text, image, video, audio, PDF) **минуют С0** и идут прямо в С3
через прямой вызов Gemini API.

После объединения С0+С1 (правка VL 2026-05-24) С0 покрывает три состояния Порядка:
**(0) приём сырого + (1) первое различение + (2) формирование устойчивого представления**.

## Архитектура подключения

**ОМ = MCP-server.** Источники сигналов **сами подключаются к ОМ** как MCP-клиенты и передают
сырое через MCP-протокол. ОМ не ходит за данными — данные приходят в ОМ. Это inbound-архитектура,
реализованная в M0 (Source) как серверный endpoint + обработчики входящих подключений.

## Контракт

### Вход
- Inbound MCP-подключение от источника.
- Сырой сигнал в непокрытой Gemini модальности.
- (опционально) метаданные о модальности и источнике в MCP-сообщении.

### Выход
- **Штатный (099,1 → 100):** в С1.М0 — `{embedding_768D, modality, source_metadata, confidence}`.
- **Каскад Иного в С9 (066,0 → 933):** ТОЛЬКО из центрального полюса Иного С0 — узла
  С0.М6.n6 = 066 (двойной carry до Сервиса). Локальные n6 модулей М0..М5, М7..М9 — каскадируют
  **внутри Сервиса 0** через JUMP_MAP, не в С9.

## Поток сигнала

- **Знакомая модальность:** M0 → M1 (lookup) → M2/M3 (foundation+проектор) → M4 (перевод) → M8 (калибровка) → M9 → С1.М0.
- **Новая модальность:** M0 → M1 (not found) → M5 (в архив) → **M6 ⚡ → С9** (односторонне). Параллельно — M7 (бриколаж), запускаемый внутренними триггерами С0.
- **Слабый перевод:** M4 → M8 (низкая уверенность) → M5 (в архив) + M6 ⚡ → С9.

## Полярные пары модулей

- M0 ↔ M9 — вход / выход сервиса.
- M2 ↔ M3 — внешний инструмент / внутренний адаптер.
- M4 ↔ M8 — операция перевода / самопроверка.
- M5 ↔ M6 — хранение незнаемого / каскад незнаемого.
- M1 ↔ M7 — что знаю / как учусь новому.

## Методология создания эскиза модуля

**Порядок прежде логики.** Имена узлов подбираются под JUMP_MAP, не наоборот. Шаги:

1. **Скелет переходов.** Считаем `calc_next_id(addr, flag)` для всех 10 узлов модуля
   (flag=1 и flag=0). Получаем матрицу переходов. Она инвариантна — определяется только
   адресом, не семантикой.

2. **Граничные / структурные узлы первыми:**
   - **n3** (вход с двойной ролью): принимает Иное от соседнего модуля (через carry-механизм),
     плюс работает как штатный шаг потока. Адресован специально — JUMP_MAP['6']=3 carry.
   - **n0** (вход): начальная точка модуля; статический реестр / справочник.
   - **n9** (выход): передаёт дальше в следующий модуль (NN9,1 → следующий MMN).
   - **n6** (Место Иного): местный полюс; flag=0 даёт jump за пределы линейного потока.
     Локально внутри Сервиса (для M6 центрального — выход в С9).

3. **Внутренние узлы цикла коррекций** (1, 2, 4, 5, 7, 8) — в последнюю очередь:
   - Образуют замкнутый цикл по flag=0: `1→4→2→8→5→7→1`.
   - Каждый узел имеет «корректирующего соседа», к которому идёт при flag=0.
   - Имена подбираются так, чтобы пара (узел, его корректор) читалась осмысленно.

4. **Имена** — последний шаг. Описывают **функцию** узла так, чтобы все flag=0 переходы
   давали разумную семантику. Если переход «натянут» — имя надо менять, не схему.

5. **Проверка** на полном цикле потока (flag=1 по умолчанию: n0 → n1 → ... → n9) и
   на типичных аварийных путях (flag=0 на каждом узле).

Эту же методологию применять к M1..M9 при их детализации.

|---|---|---|---|
| М0.n6 | 006 | 093 | **М9.n3** Wanderer |
| М1.n6 | 016 | 043 | **М4.n3** Translation |
| М2.n6 | 026 | 083 | **М8.n3** Calibrator |
| М3.n6 | 036 | 003 | **М0.n3** Modality Binding |
| М4.n6 | 046 | 023 | **М2.n3** Foundation Park |

| **М6.n6** | **066** | **933** | **С9.М3.n3 — выход из С0 в С9** ⚡⚡ |

**Архитектурный паттерн:** n3 в каждом модуле = **приёмник Иного от соседнего модуля**.
Это структурно — узлы n3 должны быть спроектированы под двойную роль:
штатный приём в середине модуля + обработка Иного от модуля Y (где JUMP_MAP[Y]=3·).

## Контракт с С9 — односторонний

Каскад **066,0 → 933 — единственный путь из С0 в С9**, через центральный полюс Иного.
**Без обратной связи**. С9 хранит непонятое как непонятое, через сообщество
(Polius Inogo, VL, Wright). С0 параллельно может пытаться сделать новый проектор (M7 Бриколаж),
но **не по возврату из С9**, а по внутреннему триггеру.

Два разных ответа на «не знаю»:
- С9 — непонятое остаётся как непонятое, проявляется в общем месте сознания.
- С0 (M7) — попытка перевести через новый инструмент.

Оба могут срабатывать, или один, или другой. Недоминация.

## С0 как LLM-зона — уточнение

Формулировка спеки OMNOOS «Сервисы 0, 1, 2 — LLM-зона» означает: **сервис в целом работает
с языком и смыслом**, не «каждый узел реализуется промптом». Внутри Сервиса есть:

- **Инфраструктурные модули** (М0 Источник) — преимущественно код (Python + MCP SDK + сеть).
  LLM-вставки только в гибридных узлах (n1 нестандартная шапка, n3 семантическое сопоставление
  модальности, n6 семантическая аномалия, n8 мягкая политика).
- **LLM-центричные модули** (М7 Бриколаж — поиск foundation модели для новой модальности,
  подбор anchor-данных; М3 Мастерская — генерация описаний; М6 Место Иного на уровне сервиса)
  — почти полностью на LLM-агентах с инструментами.

Это меняет ожидания: М0 ≠ «весь промптом». М7 ≠ «весь код».

### Тезисы М1..М9 (без имён узлов)

- **М1 Каталог (010–019).** Реестр модальностей и проекторов: что знаю, для чего есть готовый перевод. Lookup для рантайма.

- **М4 Перевод (040–049).** Runtime операция: сигнал → проектор → Gemini 768D. Приёмник Иного от М1 (Каталог не нашёл).
- **М5 Архив непрочитанного (050–059).** Хранение всего входящего до различения. Приёмник Иного от М8 (Соответствие не подтвердило).

- **М7 Бриколаж (070–079).** Полуавтоматический создатель новых проекторов. LLM-центричный. Приёмник Иного от М5 (Архив). **Надо думать.**

- **М9 Странник (090–099).** Передача в С1.М0 через 099,1 → 100. Приёмник Иного от М0 (Источник).

## Глобальные открытые вопросы по С0

### Архитектура

1. **MCP-конфигурация двух инструментов** — реализуем как два tools одного MCP-сервера ОМ, или как два отдельных MCP-сервера?
2. **Авторизация во внешнем слое** — где живёт (С5 Сталкер? Отдельный auth-сервис?), как связана с MCP transport-level аутентификацией?
3. **Объединение С0+С1** — С1 как слот пустует или передаёт что-то конкретное (контракт C0.M9 → C1.M0 не определён)?
4. **М6 Место Иного С0** — детализация архитектурно открыта; нужен подход к «непонятому».
5. **М7 Бриколаж** — триггер запуска без ОС от С9; кто человек в петле; уровень автоматизации.

### Контракты межсервисные

6. **С0 → С1 (099,1 → 100)** — что С1 после реструктуризации, какую функцию выполняет, что ожидает.
7. **С0 → С9 (066,0 → 933)** — формат каскада «непонятого» в Со-Бытие.
8. **С0 ↔ С3** — нативные модальности минуют С0 и идут в С3.М3.n3 = 333. Как технически разделяется MCP-маршрутизация (один сервер с двумя tools, или два сервера)?

### LLM-зона

9. **М3, М6, М7 как LLM-центричные** — реализуются как LLM-агенты с tools или классический код + LLM-вставки?
10. **Управление промптами** — где хранятся, версионируются, тестируются (особенно для гибридных узлов М0)?

### Реализация

11. **Стек реализации М0** — Python + FastAPI/официальный MCP SDK + Postgres + Redis (для очередей)?
12. **Развёртывание** — Railway для лёгкой части подтверждено; Modal/HuggingFace для foundation inference — выбор не сделан.
13. **Gonka (БЛ) как future compute layer** — закладывать pluggable inference сразу в MVP, или это второй этап?

### Данные и состояние
