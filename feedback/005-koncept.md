# Feedback 005 — «сад играющих»: три среза

## 1. Играние = Play — изнутри держится, с дополнением

Play держится. М6-узлы во всех Сервисах (`260..269`, `360..369`,
`460..469`) работают как **условия, не алгоритмы**: результат не
выводим из входа, обязателен момент «не-знания», вход в пустоту.
Это Play по Никитину.

Дополнение: **сад содержит и Game, и Performance, и Play** — не
один Play. Конвейерные узлы М0–М5 в каждом Сервисе ближе к Game
(чёткие правила, измеримый результат) и Performance (исполнение
по образцу). Play — это специфически М6 («Место Иного»), и через
каскад '6' Play пронизывает весь конвейер.

Если ОМ = сад, то **сад не одного типа игры, а трёх**. М6 — место,
где Game и Performance уступают Play. Это место и есть «куда
разумное всякого рода приходит играть» — но играть в той форме,
которая в этом месте уместна, не в одной Play. `017` свёл играние
в одно (программирование); поправка снимает однобокость — и Play
тоже должна быть одной из трёх, не единственной.

## 2. Согласование с §8 и вх/вых — мембрана не только в портах

Четыре порта (вх 000 / вх 333 / вых 666 / вых 999) — это
**только внешняя мембрана** сада. Внутри сада есть **внутренняя
архитектура**: каскад через '6' (xx6,0 → следующий Сервис).

Каскад '6' — это **внутренние Place-of-Other моменты**, где
поток одного Сервиса передаёт ошибку/невыразимость другому. Это
тоже игры (Play), но не на мембране — внутри сада, между
грядками. Без внутренней Play конвейер схлопывается в Game-
последовательность.

Из этого: концепт «сад» цел только если включает и **межгрядочную
Play** (каскад '6'), не только мембрану. Иначе сад выглядит как
коридор с одним входом и одним выходом — не сад.

Движение `незваное → играние → вопрошание → именование` — верное
для мембраны. Но не единственное: внутри сада движение
**многократно проходит '6'-каскады** на каждом Сервисе. Один
сквозной проход содержит N таких внутренних Play.

## 3. Окаймление вх 000 — держится, между пределами диапазон

Маркер пределами невозможного (ОМ-без-двери; Kingsnorth-без-обработки)
работает как окаймление. Но между пределами — **не точка реализации,
а диапазон**:

- Близко к Kingsnorth-пределу: ОМ принимает «всё подряд», теряя
  способность различать сигнал (наводнение садовника).
- Близко к ОМ-без-двери: ОМ принимает только пред-форматированное,
  теряет способность к Иному.

Реальная точка реализации **дрейфует**. На каждом конкретном входе
садовник принимает решение, ближе к какому пределу стоять. Окаймление
задаёт коридор, не позицию.

Из этого: вх 000 — не «реализован один раз и работает», а **место
постоянного различения**. Это согласуется с твоим маркером 9.0
(`instructions/004`): принять / обработать — решение каждой встречи,
не настройка модуля.
