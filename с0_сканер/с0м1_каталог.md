# С0.М1 — Каталог

> **Адрес:** 010–019
> **Тезис:** Реестр модальностей и проекторов: что знаю, для чего есть готовый перевод.

## Что делает

Модуль **реестра**. Хранит знание С0 о мире: какие модальности встречал, для каких есть
обученный projection head, какая foundation модель сопоставлена каждой модальности,
кто/когда/на каких anchor-данных это сделал. Lookup-эндпоинт для рантайма M4 (Перевод).

## 10 узлов — эскиз, обсудим

| # | Адрес | Имя | Тезис простыми словами |
|---|---|---|---|
| n0 | 010 | **Подключение** | Каталог принимает запросы от других модулей С0 (M3 регистрация, M4 lookup, M7 announce нового). |
| n1 | 011 | **Регистрация модальности** | Запись новой модальности в реестр: имя, ID, описание формата. |
| n2 | 012 | **Привязка foundation** | Установка связи модальность ↔ foundation модель (тип, версия, endpoint). |
| n3 | 013 | **Регистрация проектора** | Добавление projection head: путь к весам, версия, источник anchor-набора. |
| n4 | 014 | **Lookup** | Runtime: по модальности → `(foundation_ref, projector_ref, metadata)`. |
| n5 | 015 | **Версионирование** | Управление версиями проекторов: активная, история, миграции. |
| n6 | 016 | **Место Иного ⚡** | Lookup miss — модальность не известна. Каскад: в M5 (архив) + M6 (наружу в С9). |
| n7 | 017 | **Метаданные** | Кто/когда создал, на каких anchor-парах обучен, размер, лицензия, метрики калибровки. |
| n8 | 018 | **Архивация** | Снятие устаревших проекторов; пометка deprecated; миграция потребителей. |
| n9 | 019 | **Выдача** | Передача данных lookup'а в M3 (Мастерская) / M4 (Перевод) / M7 (Бриколаж). |

## Полярные пары узлов (черновик)

- n0 ↔ n9 — приём запросов / выдача ответов (стандарт I/O).
- n1 ↔ n8 — регистрация / архивация (жизненный цикл записи).
- n2 ↔ n3 — привязка foundation / привязка проектора (пара «внешний / адаптер», эхо М2↔М3 на уровне модуля).
- n4 ↔ n7 — runtime lookup / статические метаданные.
- n5 ↔ n6 — версионирование (управление) / lookup miss (полюс Иного).

## Открытые вопросы

- **Формат идентификатора модальности** — строковый ярлык (`"satellite_sentinel2"`), URI, или сложнее?
- **Где живёт хранилище** — Postgres в omnoos-backend? Yaml-конфиг? Qdrant с метаданными?
- **Версионирование (n5)** — semver, hash весов, дата + commit?
- **n6 каскад** — куда именно идёт сигнал из Lookup miss: сразу в M6, или сначала в M5 для накопления?
- **Связь с M7 Бриколажем** — Каталог триггерит M7 (по накоплению неизвестных в M5)? Или M7 сам подписан на события Каталога?

## Заметки

n6 как **Место Иного на уровне модуля М1** — это мой эскиз; не уверен в нём. Возможно правильнее
оставить М1 без своего «иного-каскада», поскольку структурный каскад незнаемого живёт в M6 сервиса.
