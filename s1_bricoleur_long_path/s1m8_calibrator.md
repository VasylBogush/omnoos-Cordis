# С1.М8 — Калибратор

> **Адрес:** 180–189
> **Стадия:** gate перед выпуском
> **Парная:** М1 Фокусировка
> **Статус:** заглушка; узлы не проработаны

## Тезис

Выходной gate качества предпонимания перед передачей в С2 (Wright):

- **уверенность узнавания** — насколько хорошо образ лёг на найденную/достроенную схему;
- **согласованность образа и предпонимания** — не противоречат ли друг другу;
- **фактичность** — нет ли явных галлюцинаций (если бриколаж в М7 сильно фантазировал);
- **полнота** — хватает ли информации для С2 baseline-аудита.

Слабое предпонимание → М5 (архив с пометкой «слабое») + М6 (Иное, для С4).

Парность с М1 (Фокусировка): фокусировка определяет, **что попадает** в обработку; калибратор определяет, **что выпускается** наружу. «Как направляем внимание / как проверяем результат».

## Контракт

### Вход (n0 = 180)
От С1.М7 (Достраивание) через `179,1 → 180` — достроенное предпонимание.
Возможный прямой путь от М4 (Поиск): если узнавание было сразу успешным (top-1 > T_high), минуя М7. **Но инвариант перехода M4 → M5 → M6 → M7 → M8 этого не позволяет напрямую.** См. открытый вопрос 1 в `s1m4_recognition_search.md`.

### Выход (n9 = 189)
В С1.М9 (Странник) через `189,1 → 190` — gate пройден.

### Каскад n6 = 186 (flag=0)
→ С1.М5.n3 = 153 — «gate не пройден, в Архив для пересмотра». Семантика: возможно, схема в архиве устарела и привела к слабому узнаванию — пометить.

### n3 = 183 — двойная роль
Приёмник Иного от **С1.М2.n6 = 126** (Гештальт не собрался → калибратор с пониженным порогом). Семантика: если на стадии гештальта была проблема, калибратор должен **знать об этом** и применить более мягкие критерии (или, наоборот, более жёсткие — TBD).

## Открытые вопросы

1. **Какие пороги?** T_confidence, T_consistency, T_factuality. Где живут (config, обучаемые, динамические)? Кто их выставляет (Wright из С2? VL? обучение)?
2. **Что значит «галлюцинация» в предпонимании?** Если М7 создал новую категорию, как проверить, что она не фантазия? Гипотеза: согласованность с близкими существующими схемами + проверка через дополнительный LLM-вызов с разной формулировкой.
3. **Слабое vs неприемлемое.** «Слабое» — пропускаем в С2 с пометкой; «неприемлемое» — отправляем в М6 (Иное → С4). Где граница?
4. **Параметризация для разных типов задач.** Один и тот же образ может быть «достаточным» для одной задачи С2 и недостаточным для другой. Калибратор должен учитывать контекст запроса? Если да — откуда он его берёт (метаданные из M0)?
5. **Минимум 4 шага.** Привычка (стандартные пороги) → ошибка (пропустили слабое; не пропустили достаточное) → жертва (отказ от текущих порогов) → наполнение (новые пороги, на основе обратной связи от С2).
