# С1 — Бриколер / Длинный путь предпонимания

> **Адресация:** 100–199 (100 узлов = 10 модулей × 10 узлов)
> **Зона:** длинный путь предподготовки к С2 — фокусировка → целостный образ → поиск узнавания / достраивание предпонимания
> **Полярная пара:** С4 (Аттрактор) — через каскад Места Иного `166,0 → 433`
> **Имя по canon:** «Бриколер / Дизайнер модальностей» (§S1 `OMNOOS_v2_SPEC.md`). Здесь не отменяется — раскрывается через метод длинного пути.
> **Статус:** рабочий эскиз — раскладка 10 модулей, узлы не проработаны
> **Дата эскиза:** 2026-05-25

## Что делает С1

Принимает от С0.М9 **текстовое описание** нестандартной модальности (через `099,1 → 100`) и проводит его через длинный путь предподготовки к С2 (систематическое строительство по чертежу).

«Длинный путь» — последовательность трёх стадий, описанных в первом разговоре с VL 2026-05-19 (`docs/Cordis.md`):

1. **Фокусировка** — внимание выделяет «это» из потока описаний и удерживает его.
2. **Целостный образ** — из разрозненных данных собирается единая вещь.
3. **Поиск узнавания / достраивание предпонимания** — образ соотносится с наличными схемами; если схема не находится — строится новая (бриколаж в смысле Леви-Стросса: «из того, что под рукой»).

«Бриколер» (canon §S1, по Леви-Строссу) — Иное снаружи; импровизация. Длинный путь = техническая форма этой импровизации в архитектуре ОМ. Не «строим проектор по чертежу» (это С2 — Wright), а **сшиваем образ и предпонимание из наличного**.

## Архитектура подключения

Внутренний модуль ОМ. **Не имеет MCP-входа извне** — принимает только от С0.М9.

- **Вход:** `099,1 → 100` от С0.М9 — `{text_description, modality, source_metadata, confidence, spec_embedding?}`.
- **Выход штатный:** `199,1 → 200` в С2.М0 — образ + достроенное предпонимание + метаданные (формат TBD, в работе с Wright по C2).
- **Выход Иного:** `166,0 → 433` в С4.М3.n3 — образ, для которого предпонимание не достроилось; материал для Перспективы (canon).
- **Вход Иного:** `766,0 → 133` от С7.М6 (Synthesis) → С1.М3.n3 — обнаруженная возможность развития предпонимания (canon: «исторический мост» от С7).

## Контракт

### Вход (M0)
- Текст + метаданные от С0.М9 через `099,1 → 100`.
- (опционально) spec_embedding как материал для подбора схем.

### Выход штатный (M9, `199,1 → 200`)
- В С2.М0 — состояние ОМ как baseline.
- **Содержание TBD:** работа на стороне Wright (С2). Гипотеза: `{image_object, recognized_schemas, extended_preunderstanding, source_chain, confidence}`.

### Каскад Иного (M6, `166,0 → 433`)
- В С4.М3.n3 — материал для Перспективы, когда наличного предпонимания не хватило (canon §S1).

## Три стадии — как раскладываются по модулям

«Поток» (стрелочки, не охваты — правка VL 2026-05-25):

- **Фокусировка** → M1 (внутри 10 узлов: отбор → захват → удержание; см. микрозазор 19 мая — валентность, тонус, подавление конкурентов, открытие окна).
- **Целостный образ** → M2 + M3 (две стороны: гештальт-группировка + бинарное связывание / константность / достраивание).
- **Поиск узнавания / достраивание предпонимания** → M4 + M7 (две стороны: чтение из наличного + создание нового).

Поддерживающие модули (структурный паттерн, как в С0):
- M0 Приём (вход) ↔ M9 Странник (выход).
- M5 Архив предпонимания (хранение схем) — читается M4, пишется M7.
- M6 Место Иного (каскад в С4).
- M8 Калибратор (gate перед выпуском в С2).

## Поток сигнала

- **Знакомая модальность (схема найдена):** M0 → M1 (фокус) → M2 (гештальт) → M3 (связывание) → M4 (поиск, нашёл в M5) → M8 (gate) → M9 → С2.М0.
- **Знакомая природа, новый вариант (бриколаж):** M0 → M1 → M2 → M3 → M4 (близкая схема в M5, но неполная) → M7 (метафорический перенос / комбинация / расщепление; запись новой схемы в M5) → M8 (gate) → M9 → С2.М0.
- **Непонятое (нет наличного предпонимания):** M0 → M1 → M2 → M3 → M4 (не нашлось) → M7 (попытка бриколажа не удалась) → **M6 ⚡ → С4.М3.n3** (`166,0 → 433`).
- **Срыв фокуса:** M1 → M6 (объект потерян до сборки образа) или M1.n6 → возврат на M4 через JUMP_MAP (см. таблицу каскадов).
- **Образ не собрался:** M2 или M3 → M6 локально через JUMP_MAP, либо возврат на M1/M2.

## Полярные пары модулей

- **M4 ↔ M5** — поиск / архив (запрос к памяти / память).

## Методология создания эскиза модуля

**«Порядок прежде логики»** (по `s0_scanner_spec.md`, методология подтверждена на М0 С0):

1. **Скелет переходов.** Считаем `calc_next_id(addr, flag)` для всех 10 узлов модуля. Получаем матрицу переходов.
2. **Граничные / структурные узлы первыми:** n3 (приёмник Иного от соседа), n0 (вход), n9 (выход), n6 (локальный полюс Иного).
3. **Внутренние узлы цикла коррекций (1, 2, 4, 5, 7, 8):** замкнутый цикл `1→4→2→8→5→7→1` по flag=0.
4. **Имена** — последний шаг. Подбираются под JUMP_MAP, чтобы flag=0 переходы давали разумную семантику.
5. **Проверка** на полном цикле и аварийных путях.

Эту же методологию применять к M0..M9 С1.

## Глобальные открытые вопросы по С1

### Архитектура

1. **Подтвердить совмещение Бриколер ↔ длинный путь.** Canon §S1 «Бриколер / Дизайнер модальностей» — раскрывается через три стадии (моё чтение) или это разные функции, и одна из них перетягивает имя?
2. **Группировка стадий по модулям.** Фокусировка = 1 модуль (M1, с 10 узлами внутри), образ = 2 модуля (M2 Гештальт + M3 Связывание), узнавание = 2 модуля (M4 Поиск + M7 Достраивание). Перевес — на стадии 2 и 3. Это правильное распределение веса, или одну из стадий нужно отдельным образом раскрыть?
3. **Третья развилка (полное / частичное / несовпадение, 19 мая).** В моём C0 — два пути (нативное → C3 / нестандарт → C0). Где живёт «частичное совпадение → внимание к остатку»? Возможно — внутри С1 как **отдельный путь сигнала** в потоке (см. узор: знакомая модальность, новый вариант → M4 → M7 → M8). Пока в моей раскладке это не отдельный модуль, а сценарий. Достаточно ли?
4. **М3.n3 как приёмник от С7.** Это canon (`766,0 → 133`). Что конкретно приходит от С7 Synthesis? Какой формат «возможности развития предпонимания»?

### Контракт с С2

5. **Формат выхода `199,1 → 200`.** Что С2.М0 (baseline-аудит) ожидает на вход? Текущее предположение `{image_object, recognized_schemas, extended_preunderstanding, ...}` — это **работа с Wright**.

### Конфликт с моим текущим эскизом С0

6. **C0.M7 «Бриколаж»** — дубль функции С1-Бриколера (canon). После того как С1 раскрыт через три стадии и включает M4+M7 как ядро бриколажа, **роль C0.M7 нуждается в пересмотре**. Варианты: переименовать (например, «Адаптер модальности» — узкая техника создания describer'а), убрать (если действительно дубль), оставить (если есть специфическая функция). Не моя текущая работа — флаг для следующей сессии работы с С0.

### Состояние Порядка

7. **С1 покрывает какие состояния Порядка?** Если С0 покрывает (0)–(2) (что VL потом снял 24.05 вечер, но логически близко), то С1 — (3)–(5)? Или это «крупный охват» (узор `feedback-too-large-poryadok-basket`)? Не подсовываю сразу — отмечаю как открытое для VL.

### Реализация

8. **Стек.** Python + LLM-вызовы (для перебора схем, метафорических переносов, генерации новых категорий). Embedding для поиска по архиву предпонимания (M4 → M5) — гипотетически Gemini Embedding 2 на тексте из С0.
9. **Хранилище M5 (Архив предпонимания).** Postgres для метаданных схем + векторный индекс? Что считается «схемой»?
