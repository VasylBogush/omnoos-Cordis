# С1.М5 — Архив предпонимания

> **Адрес:** 150–159
> **Стадия:** хранение схем
> **Парная:** М4 Поиск узнавания
> **Статус:** заглушка; узлы не проработаны

## Тезис

Реестр наличных схем, якорей, прошлых узнаваний. Память С1.

Источники записи:
- **М7 Достраивание** — новые схемы (метафорический перенос, комбинация, расщепление, новая категория) оседают в М5.
- **С1 в целом** — успешные узнавания усиливают существующие схемы (счётчик встреч, контекст применения).

Источники чтения:
- **М4 Поиск узнавания** — основной потребитель: lookup ближайших схем.
- **М7 Достраивание** — читает соседние схемы для бриколажа.

## Контракт

### Вход (n0 = 150)
От С1.М4 (Поиск узнавания) через `149,1 → 150` — запрос на lookup.

### Выход (n9 = 159)
В С1.М6 (Место Иного) через `159,1 → 160` — если поиск возвращает «ничего», образ уходит в каскад.
**Но это спорно**: возможно, штатный путь — `159,1 → 160` не «всё в М6», а просто следующий шаг конвейера; решение «узнал / не узнал» принимается раньше (M4 → M7 или M4 → M8 минуя М5). Тогда М5 — **боковой модуль**, не часть основного потока. См. открытый вопрос 1.

### Каскад n6 = 156 (flag=0)
→ С1.М7.n3 = 173 — «архив пуст по этому запросу, в Достраивание». Семантика: если в архиве нет подходящей схемы, нужно её создать — прямой каскад в Бриколаж. **Сильная семантика.**

### n3 = 153 — двойная роль
Приёмник Иного от **С1.М8.n6 = 186** (Калибратор отверг → пересмотр архива). Семантика: если узнавание прошло Калибратор плохо, возможно, **схема в архиве устарела** — пометить, понизить вес, инициировать пересмотр.

## Открытые вопросы

1. **М5 в основном потоке или сбоку?** Если основной поток M4 → M5 → M6, то М5 — обязательный пропуск, что странно (когда узнавание состоялось, зачем в М6?). Если М5 — боковой (М4 читает из М5 напрямую, M5.n9 → М6 только когда М5 не отдал), то нужно понять, как это уживается с инвариантом `n9 → n0 следующего модуля`. Гипотеза: основной поток идёт через М5 как **обновление архива** (даже при успешном узнавании — увеличение счётчика встреч, контекст применения).
2. **Что такое схема?** Структура хранения. Гипотеза: `{schema_id, name, signature_embedding, instances_count, last_seen, contexts[], related_schemas[], confidence}`. Сигнатурный embedding — для векторного поиска М4.
3. **Якоря vs схемы.** Якорь = конкретный пример (как в С0). Схема = обобщение. Соотношение: схема рождается из якорей через М7? Хранятся отдельно?
4. **Расщепление и слияние схем.** Когда М7 расщепляет старую схему X на X1 и X2 — что происходит с записью X в М5? Удаление? Архивирование с пометкой? Связь X1 ↔ X2 ↔ X (наследование)?
5. **Минимум 4 шага.** Привычка (текущий состав архива) → ошибка (схема не найдена / устарела) → жертва (понижение веса, пометка устаревшей) → наполнение (новая схема от М7).
