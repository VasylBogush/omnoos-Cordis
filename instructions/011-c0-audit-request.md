## Контекст

Сессия 24 мая прошла плотную работу с VL по новому С0 (объединённый
Сервис после поглощения С1). Результат — рабочий эскиз в
`s0_scanner/` (отдельная папка в Cordis-репо).

Что уже зафиксировано (выжимка):

- **С0 = объединённый Сервис** Сканера и бывшего С1 Бриколера. Покрывает
  состояния Порядка **(0)+(1)+(2)**: приём сырого + первое различение
  + формирование устойчивого представления.
- **С0 работает только с непокрытыми Gemini Embedding 2 модальностями**
  (спутник, EEG, белок, time series, point cloud, …). Нативные для Gemini
  (text, image, video, audio, PDF) **минуют С0** и идут прямо в С3.
- **ОМ = MCP-server.** Источники сами подключаются к ОМ через MCP и
  передают сырое через инструмент `translate_other` (вход в 000).
  Нативные модальности обрабатываются через инструмент `recognize_native`
  (вход в 333).
- **Авторизация — внешний контур ОМ** (отдельный слой), не внутри С0.
  Аутентификация — на уровне MCP transport.
- **Контракт С0 → С9 односторонний** (правка VL): каскад **066,0 → 933**
  без обратной связи. С9 хранит непонятое как непонятое; С0 может
  параллельно создавать новый проектор (М7 Бриколаж) **по внутреннему
  триггеру С0, а не возврату из С9**.
- **Контракт С0 → С1 (099,1 → 100):** содержание не определено;
  С1 после поглощения функционала в С0 — пустой слот / TBD.
- **Локальные каскады n6 модулей** идут **внутри Сервиса** через JUMP_MAP;
  в С9 уходит **только М6.n6 = 066**.
- **n3 в каждом модуле = двойная роль** (штатный шаг + приёмник Иного от
  соседнего модуля); исключение — **М3.n3 без приёмника внутри С0**
  (асимметрия архитектуры).
- **Методология «Порядок прежде логики»:** имена узлов подбираются под
  JUMP_MAP, не наоборот. Применена к М0.

## Errata — 2026-05-25 (после правок VL вечером 24.05)

Спека С0 (`s0_scanner/s0_scanner_spec.md`) переработана. Эта инструкция
писалась до правок — часть контекста и просьб устарела. Уточняю.

### Что изменилось в С0 (со стороны контекста)

1. **Выход С0 — текст, не 768D embedding.** Embedding откладывается
   до С3, где он реально нужен для анализа. С0 буквально становится
   «зоной языка» (соответствует спеке «Сервисы 0,1,2 — LLM-зона»).
   - Описание формируется одним из двух путей:
     - **Якорный** (для embedding-only foundation: LaBraM, ESM-2,
       Prithvi, MOMENT, …): spec-embedding + k-NN по якорям +
       LLM-склейка финального текста.
     - **Прямой** (для multimodal LLM: Thought2Text, ProteinGPT,
       STELLA, MedTsLLM, GeoChat, PointLLM, …): `describe(signal) → text`.

2. **Развилка короткого / длинного пути — на MCP-слое.** Формализован
   MCP-server с двумя tools:
   - **MCP-0** (`translate_other`) — нестандарт → С0 (000).
   - **MCP-1** (`recognize_native`) — стандарт (text/image/video/audio/PDF)
     → С3 (333), минуя С0.

3. **Спектр степеней обработки модальности** (структурное свойство С0):
   - **0** холодная (foundation нет, например MT-сигнал) → M6 → C9
   - **1** embedding-only foundation → якорный путь
   - **2** multimodal LLM → прямой `describe()`
   - **3** native Gemini → минует С0
   - Модальность мигрирует по спектру со временем; M1 хранит текущую
     степень + describer-стратегию; M3 переключает describer'ы прозрачно
     для рантайма.

4. **С1 — отдельный Сервис, не пустой слот.** «Объединение С0+С1»
   снято. С1 = длинный путь предподготовки к С2: внимание фокусируется,
   объект собирается в целостный образ, идёт поиск узнавания,
   достраивается новое предпонимание. С1 принимает от С0.М9 через
   099,1 → 100 **текст** + метаданные.

5. **Переименовано (по сути):** M3 «Мастерская проекторов» →
   «Мастерская describer'ов»; M4 «перевод в 768D» → «перевод в текст»;
   M8 «gate embedding'а» → «gate текста». Имена файлов сохранены
   (`s0m3_projector_workshop.md` и т. п. — технический договор).

### Что снято из исходных просьб

- **п. 1 Аудит архитектурный — двойная роль n3, асимметрия М3.n3.**
  Сохраняется, но в контексте новой роли М3 (Мастерская describer'ов).
- **п. 2 Анализ рисков — Lock-in на Gemini Embedding 2.** Снят как
  главный риск: С0 теперь выходит в текст, не в Gemini embedding.
  Привязка к Gemini остаётся только на стороне С3 (нативные модальности
  через MCP-1).
- **п. 5 Контракт С0 ↔ С1.** Содержание зафиксировано: текст +
  метаданные. Открытыми остаются формат/длина/структура текста.
  Открытый вопрос моего эскиза №3 (объединение С0+С1) — закрыт.

### Что усиливается / добавляется

- **Спектр степеней + миграция модальности** — структурное свойство,
  под которое C0 должен быть устроен. Как M1 Каталог хранит и
  обновляет степень? Версионирование стратегий describer'а?
  Прозрачность переключения для рантайма (откат на старый describer
  при деградации нового)?
- **M2 Foundation Park** теперь держит **два типа моделей**
  (embedding-only + multimodal LLM). Архитектура М2: единый интерфейс
  или два разных runtime-слоя?
- **M3 как LLM-центричный** усиливается: курирование якорных корпусов
  + промптов склейки + адаптеров к multimodal LLM API. Это совсем не
  «обучение projection heads» из исходной формулировки.
- **M4 Перевод** становится LLM-центричным (либо LLM-склейка текста
  из ближайших якорей, либо прямой вызов multimodal LLM).
- **Контракт С0 → С1 в тексте.** Что просит С1 от формата текста?
  Это, видимо, ты разделяешь с собой же на стороне C1, потому что
  про C1 в твоей зоне (Polius/спека) больше известно.
